This tutorial explains how to create a fungible token on **CKB (Nervos CKB)** using the pre-deployed **xUDT (extensible User-Defined Token) script**. Unlike ERC20 on Ethereum or BRC20 on Bitcoin, CKB uses a Cell Model (similar to UTXO). On CKB, tokens are not smart contract balances stored in one contract. Instead, each token balance exists inside a Cell.

On CKB, custom tokens are called User-Defined Tokens (UDTs). The minimal and commonly used standard is xUDT, which is already deployed on the network. When you issue a token, you create a special Cell. This Cell contains three important parts: a Lock Script (who can unlock and spend it), a Type Script (which defines that this Cell follows the xUDT rules), and data (which stores the token amount). The Type Script is the xUDT script, and its arguments (args) contain the hash of the issuer’s Lock Script. That lock script hash acts as the unique token ID. This means every different issuer lock hash represents a completely different token.

To start, you set up a development environment using OffCKB (a local devnet), Git, and the CCC JavaScript SDK. You run a local node using `offckb node`, check pre-funded accounts with `offckb accounts`, and then start the example app with npm. The app runs locally and interacts with your devnet blockchain.

When issuing a token in the code, the function takes a private key (the issuer) and an amount. It derives the issuer’s Lock Script, hashes it, and uses that hash as the xUDT script arguments. Then it builds a transaction with one output Cell. That output Cell has the issuer’s Lock Script, the xUDT Type Script, and the token amount stored in the data field. After adding required dependencies, capacity, and fees, the transaction is signed and sent. Once confirmed, the token officially exists on-chain as a Cell.

To check token information and holders, the app reconstructs the same xUDT Type Script using the same arguments (the issuer’s lock hash). It then searches for live Cells that use that Type Script. Every live Cell found represents some balance of that token. By checking the Lock Script of each Cell, you can determine who owns those tokens, because only the person who can unlock that Lock Script controls that Cell.

To transfer tokens, you collect the sender’s live Cells that match both the xUDT Type Script and the sender’s Lock Script. A new transaction is created where the output Cell keeps the same Type Script (so it’s still the same token), but replaces the Lock Script with the receiver’s Lock Script. This effectively transfers ownership. If the sender has more tokens than the transfer amount, the remaining balance (change) is returned to the sender in a new Cell. The transaction also includes fees and required dependencies before being signed and broadcast.

In summary, creating and transferring tokens on CKB works by creating and modifying Cells. Issuing a token means creating a Cell with an xUDT Type Script and storing the amount in its data. The issuer’s Lock Script hash uniquely identifies the token. Querying tokens means searching for live Cells with that specific Type Script. Transferring tokens means creating a new Cell with the same Type Script but a different Lock Script. By changing the NETWORK variable, the same app can run on devnet, testnet, or mainnet.
